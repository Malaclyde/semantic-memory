# ~~Compaction Override~~ FALSE — Research & Design

## Goal

Integrate semantic memory into OpenCode's compaction lifecycle so that:
1. The original system prompt, tool list, and recent memory chunks survive compaction verbatim
2. The standard summary sections are still generated by the compaction agent
3. The output becomes a single assistant message that provides full context continuity

---

## The Compaction Agent's Input Structure

The compaction agent receives two things:

### 1. System prompt (compaction.txt — never modified)

Source: `packages/opencode/src/agent/prompt/compaction.txt`

```
You are an anchored context summarization assistant for coding sessions.

Summarize only the conversation history you are given. The newest turns may
be kept verbatim outside your summary, so focus on the older context that
still matters for continuing the work.

If the prompt includes a <previous-summary> block, treat it as the current
anchored summary. Update it with the new history by preserving still-true
details, removing stale details, and merging in new facts.

Always follow the exact output structure requested by the user prompt.
Keep every section, preserve exact file paths and identifiers when known,
and prefer terse bullets over paragraphs.

Do not answer the conversation itself. Do not mention that you are
summarizing, compacting, or merging context. Respond in the same language
as the conversation.
```

### 2. User message (assembled by `buildPrompt()` — this is what we override)

The `buildPrompt()` function in `compaction.ts`:

```typescript
function buildPrompt(input: { previousSummary?: string; context: string[] }) {
  const anchor = input.previousSummary
    ? [
        "Update the anchored summary below using the conversation history above.",
        "Preserve still-true details, remove stale details, and merge in the new facts.",
        "<previous-summary>",
        input.previousSummary,
        "</previous-summary>",
      ].join("\n")
    : "Create a new anchored summary from the conversation history above."
  return [anchor, SUMMARY_TEMPLATE, ...input.context].join("\n\n")
}
```

The `SUMMARY_TEMPLATE`:

```
Output exactly the Markdown structure shown inside <template> and keep the section order
unchanged. Do not include the <template> tags in your response.
<template>
## Goal
- [single-sentence task summary]

## Constraints & Preferences
- [user constraints, preferences, specs, or "(none)"]

## Progress
### Done
### In Progress
### Blocked

## Key Decisions
- [decision and why, or "(none)"]

## Next Steps
- [ordered next actions or "(none)"]

## Critical Context
- [important technical facts, errors, open questions, or "(none)"]

## Relevant Files
- [file or directory path: why it matters, or "(none)"]
</template>

Rules:
- Keep every section, even when empty.
- Use terse bullets, not prose paragraphs.
- Preserve exact file paths, commands, error strings, and identifiers when known.
- Do not mention the summary process or that context was compacted.
```

The ORIGINAL system prompt from the agent's conversation is NOT included — `toModelMessage()` drops all system messages.

### 3. Tool access

The compaction agent runs with `tools: {}` — no tools available.

## The Hook: `experimental.session.compacting`

### Hook signature (from `@opencode-ai/plugin` type definitions)

```typescript
"experimental.session.compacting"?: (input: {
    sessionID: string;
}, output: {
    context: string[];
    prompt?: string;
}) => Promise<void>;
```

### How it's called in compaction.ts

```typescript
const compacting = yield* plugin.trigger(
  "experimental.session.compacting",
  { sessionID: input.sessionID },
  { context: [], prompt: undefined },
)
const nextPrompt = compacting.prompt ?? buildPrompt({ previousSummary, context: compacting.context })
```

**Semantics:**
- `output.context` — strings are APPENDED to the user message after SUMMARY_TEMPLATE
- `output.prompt` — if set, REPLACES the entire `buildPrompt()` output (anchor + SUMMARY_TEMPLATE + context are all ignored)

---

## `output.context` vs `output.prompt` — Why context is insufficient

- **Incremental compaction**: the `buildPrompt()` function handles the `<previous-summary>` block that feeds the previous compaction output back in. The hook receives no access to the previous summary.
- **Our injected strings come AFTER the template**. The compaction agent reads them but still outputs the standard template structure. The original system prompt, tools, and memories would NOT be in the output — they'd just be discarded noise in the prompt.
- **Replacing via `output.prompt`** gives us full control. We must replicate the previous-summary logic ourselves (by reading it from the session messages via the SDK client).

## Research: Can we call `client.session.messages()` from the hook?

**Verdict: SAFE — no deadlock risk.**

Evidence:
- The plugin hook fires via `plugin.trigger()`, which is a simple async iteration over hooks. No mutex, lock, or reentrancy guard is held during compaction processing.
- The HTTP call goes to a local server endpoint that does a SQLite read — fully independent of the compaction state machine.
- Pattern confirmed: `opencode-orchestrator` uses `client.session.get()` and reads state files from within hooks.

**How the client is wired** (from `packages/opencode/src/plugin/index.ts`):

```typescript
client: createOpencodeClient({
  baseUrl: "http://localhost:4096",
  headers: ServerAuth.headers(),
  fetch: async (...args) => Server.Default().app.fetch(...args),
})
```

It's a local HTTP API client. `client.session.messages()` → HTTP GET `/session/{id}/message` → SQLite read.

## Research: Does `UserMessage.system` contain the full assembled system prompt?

**Verdict: NO. It is almost always undefined.**

Evidence:
- The full system prompt is assembled at call time in `LLM.stream()` from three sources:
  1. Agent prompt (from config) OR provider-specific system prompt (anthropic.txt, etc.)
  2. Caller-passed fragments (environment block, AGENTS.md, instructions, skills)
  3. `UserMessage.system` — designed as an extension point for custom overrides
- Search for `.system =` across the entire OpenCode codebase found **zero assignments**.
- The field exists for plugin/custom-message flows. In normal operation it's undefined.

**Workaround:**
- `UserMessage.agent` IS populated — we can get the agent name
- `client.app.agents()` returns all agents with their `Info` type (name, prompt, permission, etc.)
- From the agent config, we can get the configured prompt
- Dynamic parts (AGENTS.md, environment block) are re-read on every turn and don't need preservation

**Unverified**: Can `client.app.agents()` be called from within the hook? The client is available in the closure. The pattern should work identically to `client.session.messages()`, but this hasn't been tested.

## Research: Can we get the full tool list with descriptions?

### What we CAN get:

| Source | Tool Names | Tool Descriptions |
|--------|-----------|-------------------|
| `client.app.agents()` → agent.permission | YES — which tools are allowed/denied/ask | NO |
| `client.config.get()` → config.agent | YES — configured tool permissions | NO |
| Our own plugin | YES — tools we defined | YES — we wrote them |
| Built-in tools | YES — known set | YES — can hardcode |

### What we CANNOT get:

- Tools defined in `.opencode/tools/<name>.ts` — names are visible if they appear in the permission ruleset, but descriptions are NOT exposed via the SDK
- MCP tools — names visible in permissions, descriptions not exposed
- The FULL dynamically resolved tool list (permissions × model-based filtering × feature flags)

### What this means in practice:

For our compaction output, the Available Tools section will contain:
- Tools our plugin provides (search_memory, store_memory, ...) with full descriptions
- Built-in OpenCode tools (bash, read, write, edit, grep, glob, webfetch, ...) with hardcoded descriptions
- Permission names for custom tools we don't know — listed as "`<name>` — (tool)" without description

**Unverified**: The exact shape of `Permission.Ruleset` from the agents API. We need to verify at runtime what format the permissions come in.

## Proposed `output.prompt` (custom user message for compaction agent)

```
{anchor}

Include the following sections verbatim in your output before the summary.
Do not summarize, modify, or truncate them.

## System Prompt
{system prompt from agent config}

## Available Tools
{tools with descriptions}

## Recent Memory Chunks
{5 most recent chunks from semantic memory}

Then output the session summary:

## Goal
- [single-sentence task summary]

## Constraints & Preferences
- [user constraints, preferences, specs, or "(none)"]

## Progress
### Done
- [completed work or "(none)"]
### In Progress
- [current work or "(none)"]
### Blocked
- [blockers or "(none)"]

## Key Decisions
- [decision and why, or "(none)"]

## Next Steps
- [ordered next actions or "(none)"]

## Critical Context
- [important technical facts, errors, open questions, or "(none)"]

## Relevant Files
- [file or directory path: why it matters, or "(none)"]

Rules:
- Keep every section, even when empty.
- Use terse bullets, not prose paragraphs.
- Preserve exact file paths, commands, error strings, and identifiers when known.
- Do not mention the summary process or that context was compacted.
```

Where `{anchor}` is:
- First compaction: `"Create a new anchored summary from the conversation history above."`
- Subsequent compactions: `"Update the anchored summary below using the conversation history above.\nPreserve still-true details, remove stale details, and merge in the new facts.\n<previous-summary>\n{previous summary text}\n</previous-summary>"`

The `{placeholders}` are filled by the JS hook at runtime using:
- `client.app.agents()` / `client.config.get()` for agent prompt and permissions
- Hardcoded tool descriptions for known tools
- `db.getRecentChunks(5)` for memory chunks (direct DB access from the plugin)
- Session messages API for previous summary text

## Unverified items & doubts

1. **`client.app.agents()` from within the hook** — not explicitly tested. The pattern is identical to `client.session.messages()` which is safe, but agents() goes through a different server handler path.

2. **Permission.Ruleset format at runtime** — the type definition is `{ toolName: boolean } | { "*": boolean }` but the exact JSON shape from the HTTP API needs verification.

3. **Tool count — which tools to include?** The compaction output message becomes the post-compaction context. Including ALL tools with descriptions would waste significant tokens. Unclear what the right balance is.

4. **Duplicate storage between compaction output and the re-initialized prompt** — after compaction, OpenCode re-assembles the system prompt (with AGENTS.md, environment block, etc.) fresh. Our injected "System Prompt" section in the compaction output is redundant with what the agent re-receives anyway. The value is that it's visible reference in the summary message, but the active system prompt is injected separately each turn.

5. **Session start injection** — the `experimental.chat.system.transform` hook receives `{ sessionID?: string; model: Model }`. No agent name. We can detect subagents by checking `session.parentID` but that requires `sessionID` which is optional. Unclear how to filter subagents reliably without the sessionID.

6. **`output.prompt` replaces the entire buildPrompt — including context** — if we set `output.prompt`, the user's `context` array (from any other plugins) is silently discarded. Plugins composing poorly.
